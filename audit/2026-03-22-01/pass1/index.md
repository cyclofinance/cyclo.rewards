## Findings

### F-01: Unsafe `JSON.parse` on untrusted pool data without validation [LOW]

**Location:** Line 62

```typescript
const pools: `0x${string}`[] = JSON.parse(poolsData);
```

**Description:** The pools file (`data/pools.dat`) is parsed with `JSON.parse` and the result is type-asserted as `\`0x${string}\`[]` without runtime validation. If the file contains malformed data or non-address strings, they will silently propagate through the pipeline. Unlike the blocklist (which validates each address via `parseBlocklist` -> `validateAddress`) and transfers/liquidity (which go through `parseJsonl`), pools have no structural or content validation.

**Impact:** If the pools file is corrupted or tampered with, invalid pool addresses could enter the processor, potentially causing incorrect LP position tracking or silent failures in `processLpRange()`. The type assertion gives a false sense of type safety.

**Severity:** LOW — The pools file is committed to the repo and generated by the scraper, reducing (but not eliminating) the risk of corruption. The CI determinism check provides an additional layer of defense.

### F-02: Silent swallowing of missing transfer files [LOW]

**Location:** Lines 47-50

```typescript
for (let i = 0; i < TRANSFER_FILE_COUNT; i++) {
    const transfersData = await readFile(`${DATA_DIR}/${TRANSFERS_FILE_BASE}${i + 1}.dat`, "utf8").catch(() => "");
    transfers = [...transfers, ...parseJsonl(transfersData)]
}
```

**Description:** If a transfer file fails to read (e.g., `transfers3.dat` is missing or has permission errors), the `.catch(() => "")` silently converts the error to an empty string. The pipeline will continue with incomplete transfer data without any warning. This contrasts with the liquidity, pools, and blocklist files, which all throw on read failure.

**Impact:** Missing transfer data means some accounts' balances will be under-counted, leading to incorrect reward distributions. The 0.1% tolerance check on line 171 might catch a large shortfall, but partial data loss could remain within tolerance and go undetected.

**Severity:** LOW — In practice, the CI pipeline regenerates all data files from scratch and checks for deterministic output, which mitigates this in the happy path. However, during local development or if a file is accidentally deleted, this silent failure could be confusing and lead to incorrect output being committed.

### F-03: Reward pool tolerance check allows up to 0.1% drift [INFO]

**Location:** Lines 171-173

```typescript
if (diff < 0n ? -diff > REWARD_POOL / 1000n : diff > REWARD_POOL / 1000n) {
    throw new Error(`Reward pool difference too large: ${diff}`);
}
```

**Description:** The tolerance for mismatch between total distributed rewards and the reward pool is `REWARD_POOL / 1000n`, which is 0.1%. For a 500,000-token pool, this allows up to 500 tokens of unaccounted drift. This is a design choice, not a bug — rounding in BigInt arithmetic makes exact matching impossible — but is worth documenting.

**Impact:** Informational. The tolerance exists for valid mathematical reasons (integer division truncation across many accounts). The actual drift in practice should be far smaller (dust-level).

**Severity:** INFO

### F-04: Quadratic array growth pattern for transfers [INFO]

**Location:** Lines 46-50

```typescript
let transfers: Transfer[] = []
for (let i = 0; i < TRANSFER_FILE_COUNT; i++) {
    const transfersData = await readFile(`${DATA_DIR}/${TRANSFERS_FILE_BASE}${i + 1}.dat`, "utf8").catch(() => "");
    transfers = [...transfers, ...parseJsonl(transfersData)]
}
```

**Description:** Each loop iteration creates a new array by spreading all previous transfers plus the new batch. With up to 10 files of 270,000 lines each (~2.7M transfers total), this creates progressively larger intermediate arrays, resulting in O(n^2) memory allocation. Using `transfers.push(...parseJsonl(transfersData))` or `Array.prototype.concat` would be O(n).

**Impact:** Informational. Increased memory usage and GC pressure during startup. With current data sizes this is unlikely to cause OOM, but could become relevant as data grows.

**Severity:** INFO
