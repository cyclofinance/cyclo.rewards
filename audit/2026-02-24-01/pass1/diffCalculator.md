## 2. Findings

### A03-1: Greedy/Order-Dependent Budget Allocation Produces Suboptimal Coverage

**Severity:** MEDIUM

**Location:** Lines 108-119

**Description:** The `calculateDiff` function iterates through `remainingUndistributed` in array order and greedily assigns accounts to "covered" until the budget is exhausted. Any account whose reward exceeds the remaining budget is pushed to "uncovered", even if a later, smaller account would fit. This is a variant of the knapsack problem; the greedy first-fit approach means the order of entries in `newRewards` directly determines which accounts are covered and which are not.

For example, if the remaining budget is 50 and the remaining undistributed list is `[{addr: A, reward: 45}, {addr: B, reward: 10}, {addr: C, reward: 5}]`, account A is covered (budget becomes 5), then B is uncovered (10 > 5), and C is covered (5 <= 5). If instead the list happened to be ordered `[B, C, A]`, B would be covered (budget 40), C would be covered (budget 35), and A would be covered (budget -10? no, uncovered). The point is that different orderings yield different coverage sets. Whether this matters depends on the upstream invariant that `newRewards` is sorted by descending reward (which appears to be the case from processor.ts output), but this invariant is not validated here.

**Recommendation:** Document the expected ordering invariant and/or add an assertion that `newRewards` is sorted in descending reward order before the greedy allocation, so the algorithm is deterministic and maximizes the number of covered tokens (descending order is the standard greedy heuristic for this kind of budget allocation). Alternatively, consider sorting `remainingUndistributed` explicitly within this function.

### A03-2: No Address Format Validation in `readCsv`

**Severity:** LOW

**Location:** Lines 32-35

**Description:** The `readCsv` function checks that the address field is non-empty, but does not validate that it is a valid Ethereum address (e.g., 0x-prefixed, 40 hex characters). A malformed address (e.g., a plain string, partial hex, or a string containing commas embedded in quotes) would be silently accepted. Since the CSV is generated by the system's own `processor.ts`, the risk is low for normal operation, but if any CSV is manually edited or sourced externally, invalid addresses could propagate to output files that are later used for on-chain distribution.

**Recommendation:** Add a regex check such as `/^0x[0-9a-fA-F]{40}$/` for the address field to ensure only valid Ethereum addresses are processed.

### A03-3: `readCsv` Does Not Validate the CSV Header Row

**Severity:** LOW

**Location:** Lines 14-24

**Description:** The first line of the CSV is unconditionally skipped as a "header" (the loop starts at `i = 1`), but its contents are never validated. If a CSV file is missing the header or has a different column order (e.g., `amount wei,recipient address`), the parser would silently misinterpret data rows or skip a valid data row, leading to incorrect reward calculations without any error message.

**Recommendation:** Validate that `lines[0]` matches the expected header format (`"recipient address,amount wei"`) and throw an informative error if it does not.

### A03-4: Duplicate Addresses in `newRewards` Cause Silent Double-Counting

**Severity:** MEDIUM

**Location:** Lines 77-96 (`calculateDiff`)

**Description:** When iterating the first `distributedCount` entries in `oldRewards`, the function uses `findIndex` to locate each old address in the `remainingUndistributed` clone of `newRewards`. If `newRewards` contains duplicate addresses (which `readCsv` allows -- confirmed by test at line 138 of the test file), `findIndex` only removes the first occurrence via `splice`. The second occurrence remains in `remainingUndistributed` and will be processed as an "uncovered" or "covered" account, leading to double-counting of rewards for that address.

More critically, in the covered/uncovered loop (lines 108-119), duplicate addresses would each consume budget independently, meaning the same address could appear multiple times in the output CSVs. If these CSVs are fed into an on-chain distribution tool, the same address could receive rewards multiple times.

The test suite (`diffCalculatorOutput.test.ts`) does check for no duplicate addresses in the actual CSV outputs, but the `calculateDiff` function itself does not enforce this invariant.

**Recommendation:** Either deduplicate `newRewards` at the start of `calculateDiff` (summing rewards for the same address), or add an explicit check that rejects duplicate addresses with an error. The same applies to `oldRewards`.

### A03-6: `main()` Executes on Module Import -- No Guard Against Accidental Execution

**Severity:** MEDIUM

**Location:** Lines 180-181

**Description:** The `main()` function is invoked unconditionally at module scope (line 181). This means any module that imports from `diffCalculator.ts` (even for just the `readCsv` or `calculateDiff` functions) will trigger file I/O: reading two specific CSV files from disk and writing three output files. The test file works around this by mocking `fs` before import, but this is fragile. If a future module imports `calculateDiff` for reuse, it would unexpectedly execute the full Dec 2025 rewards pipeline and overwrite output files.

**Recommendation:** Guard the execution behind a check such as `if (import.meta.url === ...)` or move it to a separate entry-point script. This is both a correctness and a safety concern -- an accidental import could silently overwrite output CSVs with stale or incorrect data.

### A03-7: Hardcoded File Paths and `DISTRIBUTED_COUNT` Reduce Auditability

**Severity:** LOW

**Location:** Lines 4, 136-137, 148, 157, 166

**Description:** The `main()` function hardcodes:
- Input file paths: `./output/rewards-51504517-52994045.csv` and `./output/rewards-51504517-52994045-old.csv`
- Output file paths: three CSVs in `output/`
- `DISTRIBUTED_COUNT = 100` as a module-level constant

These are all specific to the Dec 2025 epoch. While this is a one-time script for a specific reconciliation, the lack of parameterization means it cannot be reused for future epochs without modification. The hardcoded `100` has no accompanying documentation of why exactly 100 accounts were distributed.

**Recommendation:** Add a comment explaining the provenance of `DISTRIBUTED_COUNT = 100` (e.g., "first 100 rows of old rewards were distributed on-chain in tx 0x..."). Consider parameterizing file paths via command-line arguments or environment variables for future reuse.

### A03-8: Negative Reward Values Are Accepted Without Validation

**Severity:** LOW

**Location:** Line 40 (`readCsv`), confirmed by test at line 122

**Description:** `BigInt(rewardStr)` will successfully parse negative numbers (e.g., `-1000`). A negative reward in the input would cause `calculateDiff` to behave unexpectedly: it would increase `remainingRewardsDiff` instead of decreasing it (line 117), effectively expanding the budget. In the old rewards, a negative value would decrease `totalAlreadyPaid`, again expanding the apparent remaining budget.

While the test suite's `diffCalculatorOutput.test.ts` checks that actual CSVs have no negative values, the `readCsv` and `calculateDiff` functions themselves accept them without complaint.

**Recommendation:** Add validation in `readCsv` to reject negative reward values, or at minimum in `calculateDiff` to reject entries with negative rewards, since negative rewards are semantically nonsensical for a distribution calculation.

### A03-9: Potential Underflow in `main()` Console Output

**Severity:** INFO

**Location:** Line 177

**Description:** The expression `result.totalUnderpaid + result.totalRemainingUncovered - result.remainingRewardsDiff` computes the "EXTRA needed to complete all payments." Since these are BigInt values, this subtraction can produce a negative result (which is fine for BigInt arithmetic -- no underflow/overflow). However, a negative result would mean the pool has surplus, and the log message "Total EXTRA needed" would be misleading (it would print a negative number). This is a cosmetic/informational issue only, as the value is only logged and not used in any output file.

No action needed for security, but the log message could be improved for clarity.

### A03-10: No Validation That `distributedCount` Is a Non-Negative Integer

**Severity:** LOW

**Location:** Line 61-69

**Description:** The `distributedCount` parameter is typed as `number`, which in JavaScript/TypeScript can be a float, `NaN`, `Infinity`, or negative. While there is a check that `distributedCount > oldRewards.length` (line 67), there is no check for:
- Negative values (a negative `distributedCount` would skip the first `for` loop entirely, as `0 < negative` is false, effectively treating it as 0)
- Non-integer values (a float like `1.5` would cause the loop `i < 1.5` to run once with `i = 0`, silently truncating)
- `NaN` (would skip the loop entirely)

Since `distributedCount` is always called with the constant `100`, this is not exploitable in practice, but it is a defensive programming gap.

**Recommendation:** Add `if (!Number.isInteger(distributedCount) || distributedCount < 0) throw new Error(...)` at the start of `calculateDiff`.

### A03-11: Overpaid Accounts Are Silently Ignored (No Clawback Tracking)

**Severity:** INFO

**Location:** Lines 84-93

**Description:** When an old distributed account has a new reward less than or equal to what was already paid (`diff <= 0`), the account is simply spliced out of `remainingUndistributed` with no further action. The overpayment amount (`oldItem.reward - newItem.reward`) is not tracked or reported anywhere. This means:
- The overpayment permanently reduces the effective pool available for other accounts.
- There is no output CSV or log line showing how much was overpaid and to whom.

This may be by design (no clawback mechanism), but it reduces transparency of the reconciliation.

**Recommendation:** Track overpaid amounts and either log them or include them in the output for auditability. At minimum, add a `totalOverpaid` field to `DiffResult`.
